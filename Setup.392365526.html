Setup

Introduction

This document describes setting up a developer machine for BMP.

There are two ways to set up development; local, and development container. The local setup is perhaps easier, and at least a little lighter on resources, while the development container setup makes it easier to have a uniform development environment across different machines.

Local development setup

Prerequisites

Project

We assume that you have created a corporater directory somewhere which will contain your projects, and that your terminal is in this directory.

Clone the repository

$ git clone git@github.com:corporater/bmp.bmp.git
$ cd bmp.bmp

You should be at the root of the cloned repository for the rest of the steps.

Set environment variables

Download the file credentials.txt.

Linux and MacOS 
bash -c 'source credentials.txt && for v in USR PSW TOKEN; do var="ARTIFACTORY_$v"; echo "export $var=${!var}"; done' | tee -a ~/.profile >> ~/.zshrc
> Note if using zsh replace .profile with .zprofile above > It is important for IDEs to have access to these environment variables for a good developer experience. The GNOME display manager will source the `$HOME/.profile` file, so if we export environment variables in this file, any process started by GNOME (like IDEs), will inherit those variables. Windows In Windows, you need to set then environment variables from `credentials.txt` in the Windows Control Panel, System, Advanced system settings...

Tool settings

Linux and MacOS Copy `.circleci/settings.xml` to `.m2/` folder in home dir: 
$ mkdir ~/.m2
$ cp .circleci/settings.xml ~/.m2
Windows 
mkdir %userprofile%\.m2
copy .circleci\settings.xml %userprofile%\.m2
Note: After the introduction of pnpm, we've seen issues with using environment variable in npmrc file. If you see the error message "err_INVALID_CHAR Invalid character in header content ["authorization"]", try replacing the environment variable ${ARTIFACTORY_TOKEN} in npmrc file with the actual token. ### Run maven install To validate your development environment, run 
$ mvn clean install -DskipTests
--- # Development container setup The main reason to use a development container is consistency. A development container will be the same no matter what machine it is running on, and will use the designated dependency versions, ensuring consistent builds and behavior. It is also a lot easier to set up a new development machine. The only prerequisite is docker. It also makes it easy to work on projects requiring different versions of the same dependency. ## Prerequisites To use the development container, you will need to install docker. Follow the guides to [get docker](https://docs.docker.com/get-docker/) for your OS. Visual Studio Code is an optional dependency, but using VS Code with the [_Remote - Containers_](https://marketplace.visualstudio.com/items?itemName=ms-vscode-remote.remote-containers) plugin makes the experience almost seamless. [Download](https://code.visualstudio.com/download) and [install](https://code.visualstudio.com/docs/setup/setup-overview) VS Code, then install (`Ctrl+Shift+X`) the _Remote - Containers_ plugin. ### Windows To have a good experience using the development container on Windows, you will need to [install WSL](https://docs.microsoft.com/en-us/windows/wsl/install). Once installed, start a terminal running in WSL and follow along... ## Project ### Clone the repository We assume that you have created a `corporater` directory somewhere which will contain your projects, and that your terminal is in this directory. ### Clone the repository 
$ git clone git@github.com:corporater/bmp.bmp.git
$ cd bmp.bmp
You should be at the root of the cloned repository for the rest of the steps. ### Set environment variables Download the file [`credentials.txt`](https://corporaterglobal.sharepoint.com/sites/CorporaterDevelopment/Delte%20dokumenter/devs/Dev%20environment%20setup/credentials.txt), then execute the following command: 
$ bash -c 'source credentials.txt && for v in USR PSW TOKEN; do var="ARTIFACTORY_$v"; echo "export $var=${!var}"; done' >> ~/.zshrc
$ source ~/.zshrc
If you're running in WSL on Windows in the default Ubuntu Linux distribution and the default shell, change `~/.zshrc` to `~/.bashrc`. ### Use Visual Studio code with devcontainer Open VS Code on the root of your repository clone. Either - start VS Code, select _File -> Open Folder..._ (`Ctrl+K Ctrl+O`) and select your directory, or - execute ```sh $ code . ``` when standing in your project catalog. When VS Code opens on your project (and you _did_ install the _Remote - Containers_ plugin, didn't you?), you should get a notification _Reopen in container_ button in the lower right corner. Press the button, wait for docker to build the development container image and start it, and then VS Code will open on the project again. Only this time it will run inside the development container, only running the GUI client on the host machine. If you open the terminal (`` Ctrl+Shift+` `` or menu _Terminal -> New Terminal_) you can run commands (like `mvn clean install`) inside the container. #### Windows Using VS Code with devcontainer in WSL on Windows is slightly more complicated. You will need to install the [_Remote - WSL_](https://marketplace.visualstudio.com/items?itemName=ms-vscode-remote.remote-wsl) plugin, then open VS Code in WSL (`Ctrl+Shift+P` then search for the _Remote-WSL: New Window_ command, or press the green remote icon in the lower left corner). In the new WSL window, you should be able to open your project and start the devcontainer. ### Use terminal with devcontainer It is also possible to start the development container without using VS Code. There is a shell script in the `.devcontainer` folder that does the trick: 
$ ./.devcontainer/devcontainer.sh
This will open a zsh in your development container. ### Notes on the BMP devcontainer #### Caching The BMP devcontainer will cache maven and NPM downloads in the `.devcontainer/remote/m2` and `.devcontainer/remote/npm` directories. The zsh history file is also cached. If you run BMP in the container (see below), any corporater files will be cached in the `.devcontainer/remote/corporater` directory. #### Running BMP in devcontainer There are predefined VS Code launch configurations for _Control Center_, _Configuration Studio_, and _BMP Frontend_. There are also launch scripts that can be used from the terminal in the `.devcontainer/launch` directory. The _BMP Frontend_ launch configuration should be usable on any platform. The other launch configurations require a GUI, which is not available in the development container. If you are running on Linux, you can still run these programs, since the XServer socket has been mapped into the devcontainer, as is the `DISPLAY` environment variable. It should work out of the box. On MacOS and Windows, this will not work unless you are running a X Server, which is way out of scope for this documentation. But you should be able to run the programs from your host machine if you install java, by using the launcher scripts. --- # Development ## IDE Setup ### IntelliJ Applications can be launched using existing run configurations.
