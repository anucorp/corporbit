Frontend Roadmap

Corporater frontend development roadmap

This document summarizes the current frontend state together with best practices and has been discussed with Gryffindor. Anyone is welcome to comment and contribute.

Current state

  • We make use of Apollo GraphQL client.
  • We also use Redux to manage the global state.
  • For testing we have React and Enzyme.
  • To separate UI concerns we use HOCs and Recompose.
  • We use TypeScript.
  • We support browsers defined in .browserslistrc.

Frontend challenges

  • The project structure drifted away from the original model.
  • Application context is overly complicated.
  • GraphQL is tailored to the frontend, not to the business model.
  • Lack of server to client communication. This leads to stale data displayed in UI after editing objects. Not urgent, but could open some new possibilities for users and developers.
  • Don't use setImmediate. This is deprecated and has worked only in node and IE11. IE11-specific code should be removed.
  • Remove dynamic mock. Use partial types.
  • Incomplete typings and hard to type HOCs.
  • Whitebox tests can be brittle and prevent us from refactoring.
  • Enzyme is losing popularity. And react-testing-library has become a new standard. It is present in the official create-react-app boilerplate.
  • HOCs are hard to type. It is not always intuitive. Recompose is hard to type and has been discontinued.

Misc

  • Redux and React-Redux are globally mocked by default and return undefined in tests.
  • End to end tests don't use the development server by default in cypress.json.
  • Be aware how Apollo's cache keys are resolved in the code.
  • Creating Redux store has been changed recently.

Adapting for the future

Warnings reported by ESLint and React

  • Newly added ESLint reports many errors that should be fixed gradually.
  • We have the newest TS version, but we don't fully use all its capabilities. ESLint warns about untyped variables. Aim for 100% type coverage. Use inferred types inside modules and explicit types for module boundaries.. No need to explicitly type everything, TS can infer types for us.
  • Ensure React 17 forward compatibility by addressing React console errors.

React

  • Avoid prop drilling. Use React Context and Hooks.
  • Once popular Recompose has been deprecated by the author in favor of hooks. Replace HOCs with hooks.
  • We are working towards upgrading to React 17.

Testing

  • White box tests in Enzyme contain a lot of mocks and work on low level, which can make change difficult. Use testing-library to test the usage, not the structure.
  • Testing async functionalities using enzyme is not robust. Use react-testing-library which should have better support for that.

GraphQL

  • GraphQL is tailored to the frontend, not to the business model. This makes the number of fetchers grow proportionately to the UI components and limits reusability. Write GraphQL to model the domain, not the UI.
  • Mocking GraphQL is difficult. Use some recipes to simplify that: [[React Testing Library and Apollo]]. Try mocking at the service worker level.
  • Dynamic mock is not needed when types are applied correctly. Remove dynamic mock.
  • Use ramda instead of lodash. It is similar, but has functional programming capabilities. We would like to remove lodash from package.json.
