Security Recommendations

Security recommendations

All data coming from the web exposes security threat and should be treated thoroughly.

Here are some common issues and mitigation recommendations:

  • Unauthorized access to servlets.

All servlets should be protected with a filter to prevent unauthorized access (whenever it is possible).

For example, authFilter can be used (com.corporater.run.AuthFilter).

  • CRLF Injection (Carriage Return - \r, Line Feed - \n) in headers.

All user input which can end-up in headers should be sanitized.

All headers are seperated with new lines, hence, an attacker can inject new lines with additional headers which will be sent in request.

Use Encode.forJava(value) from ESAPI to sanitize a header value.

Before: msg.setHeader("user", userId);, after: msg.setHeader("user", Encode.forJava(userId));

  • Improper Output Neutralization for Logs.

All user input which can end-up in logs should be sanitized.

An attacker is able to insert new lines in logs and fake them.

Use ESAPI logger when logging values received from the Web.

Initiate logger private static final Logger LOGGER = ESAPI.getLogger(ClassName.class.getName());,

log info message LOGGER.info(EVENT_SUCCESS, message); or an error message LOGGER.error(EVENT_FAILURE, ex.getMessage(), ex);.

Before LOG.info("User logged in to workspace: " + req.getParameter("ws"));,

after ESAPI_LOGGER.info(Logger.EVENT_SUCCESS, "User logged in to workspace: " + req.getParameter("ws"));

  • Use of Hard-coded Password.

Do not store hard-coded credentials in code, load secrets from property/configuration files instead.

  • Cross-Site Scripting (XSS)..

Improper Neutralization of Script-Related HTML Tags in a Web Page.

All user input which can end-up on Web should be sanitized.

An attacker is able to insert scripts through HTML.

We use OWASP Antisamy for removing scripts: String sanitizedInput = new HtmlText(input).noxss().toString();.

Alternatively, it is possible to encode the input with OWASP encoder: Encode.forHtml(input);.

Often user input is returned to Web as part of an error message.

In this case, log full error message and return hard-coded error message to Web.

  • Generation of Error Message Containing Sensitive Information.

Recommendation from Veracode: “Ensure that only generic error messages are returned to the end user that do not reveal any additional details.”

Log full message, return static message without sensitive information to Web.

Before (exception might contain ids or some other sensitive data) res.sendError(HttpServletResponse.SC_UNAUTHORIZED, e.getLocalizedMessage());,

after LOGGER.error(Logger.EVENT_FAILURE, e.getMessage(), e); res.sendError(HttpServletResponse.SC_UNAUTHORIZED, "Invalid Access Token");.

  • Directory Traversal.

Validate file path before opening the file.

Use IOFactory.filepathValid(dirOrFileName); to validate the file path.

Before File file = new File(path);,

after File file = new File(IOFactory.filepathValid(path));.

  • Deserialization of Untrusted Data.

Validate data coming from untrusted sources.

Add whitelist of classes which can be deserialized.

One of the solutions is to use ValidatingObjectInputStream from Apache Commons IO (https://cheatsheetseries.owasp.org/cheatsheets/Deserialization_Cheat_Sheet.html#java).

Before: try (ObjectInputStream ois = new ObjectInputStream(req.getInputStream())) { Set<IntegrationCommand<?>> out = new LinkedHashSet<>(); int count = ois.readInt(); for (int i = 0; i < count; i++) { try { out.add((IntegrationCommand<?>) ois.readObject()); } catch (Exception e) { throw new IllegalStateException(e); } } return SeqFactory.from(out); } After: try (ValidatingObjectInputStream vois = new ValidatingObjectInputStream(req.getInputStream())) { vois.accept("java.lang.*", "java.util.*", "com.corporater.*"); Set<IntegrationCommand<?>> out = new LinkedHashSet<>(); int count = vois.readInt(); for (int i = 0; i < count; i++) { try { out.add((IntegrationCommand<?>) vois.readObject()); } catch (Exception e) { throw new IllegalStateException(e); } } return SeqFactory.from(out); }

  • Improper Resource Shutdown or Release.

Use "try-with-resources" statement that prevents improper resource shutdown or release issues.

  • Vulnerabilities from dependencies.

Prior to importing new dependencies evaluate the security risk. Maven Repository is now displaying known vulnerabilities for each dependency.

P.S. We are not posting specific links, there are various sources which can provide extended description of issues.
Popular ones: CWE (cwe.mitre.org), OWASP (owasp.org) and Veracode (veracode.com).
